# 제품 요구사항 & 기능 상세 명세 (PRD) — skinmate

> 대상 독자: 구현 LLM/개발자. WBS의 각 작업이 "무엇을 왜, 어떤 흐름·규칙·예외로" 만들어야 하는지의 근거.
> 근거: [deep-interview-skinmate.md](../deep-interview-skinmate.md)(요구·AC), [skinmate-consensus-plan.md](../skinmate-consensus-plan.md)(설계).
> AC-ID는 [ACCEPTANCE-TESTING.md](ACCEPTANCE-TESTING.md)와 1:1 대응.

## 0. 제품 한 줄 정의 & 목적

**다중 사용자 각각에게 개인화된 대화형 화장품 추천을 제공한다.** 세 컨텍스트 소스를 융합한다: ① 사용자별 격리 **기억**(선호·회피·피부상태), ② 실데이터 기반 **구조화 지식 + 문서 임베딩**(성분·제품·피부관리 프로즈), ③ 사용자별 **관계 그래프**(2+hop 순회로 근거 생성).

**왜 이렇게 만드나:** 단순 벡터 검색만으로는 "왜 이 제품인가"를 다단계 근거(고민→성분→제품, 회피성분→대체성분)로 설명하지 못한다. 그래프 순회가 "풍부한 근거"라는 제품 차별점을 만든다. 기억은 같은 질문이라도 사용자마다 다른 답을 내는 개인화의 원천이다.

## 1. 공통 런타임 흐름 (모든 턴의 골격)

모든 대화 턴은 **읽기 → 응답 → 쓰기** 구조가 동일하다.

```
사용자 입력
  → (B) 의도 분류: 정보진술 / 추천요청(구체·모호)
  → [추천요청이면] (A) Retrieval 융합 → 하드필터 → 제형 soft-rank → (B) 근거 생성 → 응답
  → [정보진술이면] (B) 가벼운 확인 응답
  → 응답 반환
  → (B) 응답 후: fact 추출 → CRUD 판정 → (writer) 단일 트랜잭션 원자 저장(표+벡터+그래프)
```

- **정보진술 턴**은 read(추천)가 비고 write가 주력.
- **추천 턴**은 read가 주력, write는 새 사실이 있을 때만.
- 두 시나리오의 구체 흐름은 §8에.

## 2. F1 — 기억 (Memory) [담당 B]

### 목표
사용자가 흘린 **중요한 도메인 사실**(피부타입·회피/선호 성분·제형·브랜드·고민)만 골라 저장하고, **자주·최근** 언급된 것을 우선순위 상위로 올려 추천에 반영한다. 일상 잡담은 저장하지 않는다.

### 사용자 시나리오
1. 사용자가 "레티놀 써봤는데 자극돼서 안 맞았어요"라고 말한다.
2. 응답 후, fact 추출기가 `{fact_type: avoid_ingredient, value: 레티놀, reason: 자극}`을 뽑는다.
3. 중요도 필터가 "도메인 사실"로 판정 → 저장 대상.
4. 기존 기억과 유사도 비교(같은 사용자 기억 공간) → 중복 아님 → **add**.
5. writer가 memories 행 + 감사로그 + 그래프 AVOIDS 엣지를 한 트랜잭션에 저장.
6. 다음 턴부터 이 사실이 추천에 반영된다.

### 비즈니스 로직
- **CRUD 판정(LLM)**:
  - `add`: 새 슬롯의 사실.
  - `update`: **같은 슬롯**의 값 변경(예: "이제 레티놀 괜찮아졌어" → 회피 해제/전환). 동일 슬롯 최신값으로.
  - `delete`: 철회·무효화(soft-delete + 감사로그).
  - `no-op`: 이미 있는 사실의 중복 언급 → 저장 안 함, 단 **frequency +1 / last_seen 갱신**(가중치 상승).
- **중요도 필터**: 피부·성분·제형·브랜드·고민·루틴 = 저장. "오늘 피곤해" 같은 비도메인 = 미저장.
- **가중치**: `effective_weight = base_weight × exp(-λ × Δdays)`, `λ=0.05/day`(≈14일 반감기). `base_weight`는 언급 빈도로 증가, 조회 시 시간감쇠 계산.
- **비파괴 공존(중요)**: 서로 충돌하지 않는 새 사실이 들어와도 기존 참인 사실을 **삭제·덮어쓰지 않는다**(둘 다 유지).
- `fact_type` enum: `skin_type · avoid_ingredient · prefer_ingredient · avoid_brand · prefer_brand · has_concern · other`.

### 예외 처리
- LLM 추출/판정 JSON 파싱 실패 → **1회 재시도 → 그래도 실패면 no-op**(그 턴은 저장 스킵) + ERROR 로그. 사용자 응답은 이미 나갔으므로 대화는 계속.
- 모순 감지("예전엔 안 맞았는데 지금 괜찮아") → `update`로 슬롯 최신화, 이전 값은 감사로그에 보존.
- delete는 **절대 하드 삭제 금지**(오판정 복구 위해 soft-delete).
- 성분명을 성분 테이블 ID로 해석 못 함 → 관계형 memories에는 원문 저장, 그래프 엣지만 skip.

## 3. F2 — 구조화 지식 & 하드필터 (Structured Knowledge) [담당 A]

### 목표
실데이터로 성분(등급·효과·분류)·제품·성분-제품 관계를 관계형 테이블에 적재하고, **회피 성분이 절대 추천에 안 들어가는 하드필터**의 근거(진실원)를 제공한다.

### 사용자 시나리오
1. 사용자가 레티놀 회피로 등록되어 있다.
2. 추천 후보 제품 목록이 만들어진다.
3. 하드필터가 `product_ingredients`를 조회해 레티놀을 포함한 제품을 **전량 제거**한다.
4. 남은 제품만 다음 단계(제형 랭킹)로 넘어간다.

### 비즈니스 로직
- `product_ingredients`(관계형)가 **containment의 진실원**. 하드필터는 여기에만 건다(그래프 아님).
- 성분 canonical key = INCI 우선, 없으면 한글명 정규화. 중복은 병합.
- 성분→제품 조회(AC-D2)는 canonical key로 정확 집합 반환.

### 예외 처리
- 회피 성분이 어떤 제품에도 없음 → 필터는 전량 통과(정상).
- 하드필터 결과 후보가 0개 → 추천 대신 "조건을 만족하는 제품을 못 찾음 + 대체 방향 제안"(퍼널로 완화). **빈 추천을 근거 없이 만들지 않는다.**
- 성분 데이터 누락(제품에 성분 정보 없음) → 그 제품은 **하드필터 안전을 위해 보수적으로 제외**하고 로그(회피 성분 포함 여부를 확신 못 하므로).

## 4. F3 — 문서 임베딩 / RAG (Documents) [담당 A]

### 목표
"피부관리 방식" 프로즈와 제품 설명을 임베딩으로 저장해 유사도 검색을 제공하고, **제형(에멀전/오일/젤/끈적임) 신호를 제품 설명 임베딩으로 근사**한다(별도 구조화 태그 없음).

### 사용자 시나리오
1. 사용자가 "끈적이지 않는 보습"을 원한다.
2. 벡터 검색이 "가벼운 텍스처·에멀전·수분" 관련 문서·제품을 유사도 상위로 반환한다.
3. 이 결과가 제형 soft-ranking의 입력이 된다.

### 비즈니스 로직
- 임베딩: bge-m3, `D_DOC=1024`, 다국어. 대상: `documents`(프로즈) + `products.description`.
- 모든 벡터 행에 `embedding_model_id` 기록.
- **AC-D1 제형토큰 게이트(measure-first)**: 적재 후 제품 설명의 제형 토큰 커버리지를 **실측**. ≥60% 커버리지 + 어휘 다양성 확보면 통과. 미달 시 손 큐레이션 보강(= 제형 테스트 fixture 겸용). 토큰 사전·임계치 세부는 실측 후 확정.

### 예외 처리
- 임베딩 모델 로딩 실패(로컬) → 앱 기동 실패로 명확히(조용한 저품질 금지). `EMBEDDER_MODE` 폴백 안내.
- 크롤 텍스트에 제형 토큰 희박 → 게이트 미달 리포트 + 보강 트리거(1A.3).
- 유사도 결과 비어있음 → 추천은 그래프·기억 경로로 폴백(빈 답 금지).

## 5. F4 — 관계 그래프 & 순회 (Graph, AGE) [담당 A + 다리 B]

### 목표
성분→고민(TREATS), 성분→성분(HELPS/CONFLICTS), 제품→성분(CONTAINS), 사용자→회피/선호/고민(개인) 관계를 그래프로 저장하고 **2+hop 순회**로 추천 근거를 만든다. 순회 경로 자체가 사용자에게 보이는 근거가 된다. (계절은 개인 맥락 — memories.season, HAS_CONCERN 엣지 프로퍼티)

### 사용자 시나리오 (핵심 순회, AC-G2)
1. 사용자: 민감성 + 레티놀 회피 등록.
2. 순회: `(:User)-[:AVOIDS]->(레티놀)` → `(:Product)-[:CONTAINS]->(레티놀)` 제외 → `(:Ingredient)-[:HELPS/TREATS]->(대체 성분)` → 그 성분 포함 대체 제품.
3. 이 경로가 "레티놀을 피하고 대신 X 성분이 든 Y를 권함"이라는 근거 문장으로 변환된다.

### 비즈니스 로직
- **엣지 원천 두 갈래** (자세히는 [DATA-MODEL.md](DATA-MODEL.md) §2):
  - `CONTAINS`(product_ingredients 투영), 개인 엣지 `HAS_CONCERN{season?}`·`AVOIDS/PREFERS`(memories 투영, `user_scope` 스탬프) — **관계형에서 projection**, idempotent 재실행.
  - `TREATS/AGGRAVATES`·`HELPS/CONFLICTS` — **그래프 네이티브**(테이블 없음, 인제스트가 문서·크롤에서 추출·적재). 재구축은 재-인제스트.
- **choke 단일 관문**: 모든 AGE 접근은 `age_exec(conn, user_scope, cypher, params)`. 개인 라벨엔 scope 강제, 전역 read만 scope=None 허용.
- **제형은 그래프 노드가 아니다**(임베딩 전용). AVOIDS/PREFERS 대상도 성분·브랜드뿐, 제형 제외.

### F4-bridge — 기억→그래프 다리 [담당 B, ⭐5]
- memories의 `fact_type ∈ {avoid/prefer_ingredient, avoid/prefer_brand, has_concern}` 행을 **writer의 같은 트랜잭션 안에서** 개인 AGE 엣지로 투영.
- 성분명→`ingredient_id` 해석: INCI lookup → 한/영명 fallback → 실패 시 엣지 skip + WARN.
- 기억 soft-delete 시 대응 엣지도 같은 트랜잭션에서 제거.

### 예외 처리
- 순회 결과 경로 없음 → 벡터·기억 기반 추천으로 폴백 + "그래프 근거 없음"을 근거에 과장하지 않음.
- 잘못된 scope로 개인 엣지 접근 시도 → choke가 0건 반환(누수 금지). 이건 AC-G3 테스트로 보증.
- projection 재실행 시 중복 엣지 → upsert 의미로 idempotent(중복 생성 금지).

## 6. F5 — 저장 동기화 / 원자 저장 (Storage Sync) [담당 B, ⭐4]

### 목표
기억 쓰기(관계형 + 벡터 + 그래프)를 **한 트랜잭션으로 원자적**으로 처리해 부분 저장을 원천 차단한다. 큐·워커·drain 없음(동기).

### 사용자 시나리오
1. 사용자가 새 사실을 흘린다 → 응답이 먼저 나간다.
2. writer가 한 커넥션에서 `BEGIN` → memories INSERT → (필요시)벡터 write → 그래프 AVOIDS 엣지 write(choke) → `COMMIT`.
3. 도중 그래프 write가 실패하면 → `ROLLBACK` → memories 행까지 **전부 취소** → 부분 저장 0.
4. 성공하면 다음 턴 검색이 **즉시** 이 사실을 본다(staleness 없음).

### 비즈니스 로직
- **한 커넥션·한 트랜잭션·autocommit 금지**(불변식).
- 응답 생성 **후** 실행(첫 토큰 지연·답변 품질 보호).
- 같은 사용자 동시 요청은 **per-user advisory lock**으로 직렬화.
- AC-S1 범위 = **턴당 기억 쓰기 경로만**. 대량 인제스트·전역 projection은 별도 배치(자체 idempotent 재구성).

### 예외 처리
- 트랜잭션 예외 → ROLLBACK + 그 턴은 "저장 실패" 로그(응답은 이미 커밋된 상태에서 생성됨).
- advisory lock 대기 타임아웃 → 재시도 1회 후 실패 로그(무한 대기 금지).
- 커밋 후 크래시 → 이미 영속화(원자성 보장 지점은 COMMIT).

## 7. F6 — 대화형 추천 (Chat/Recommendation) [담당 B]

### 목표
**적응형 산출물**: 질문이 구체적이면 바로 제품명+근거, 모호하면 조언→대화로 좁혀 최종 제품에 도달. 근거는 **실제 그래프 경로·기억과 일치**(환각 금지).

### 사용자 시나리오 A — 구체 질문
1. "가을이라 건조한데 끈적한 오일 말고 에멀전으로 보습 확실한 거 추천해줘."
2. 라우팅=구체 → 즉시 Retrieval 융합 → 하드필터 → 제형 soft-rank(에멀전>오일) → 근거 생성.
3. 응답: "○○ 에멀전 — 건조 고민(→보습 성분 TREATS 경로) + 오일 회피(기억) + 가을 맥락 반영."

### 사용자 시나리오 B — 모호 질문
1. "요즘 피부가 별로인데 뭐 쓰면 좋을까?"
2. 라우팅=모호 → 방향 조언 + **좁히기 질문**("어떤 고민이 제일 크세요? 건조/트러블/민감?").
3. 사용자 응답으로 퍼널 상태 갱신 → 반복 → 조건이 충분해지면 구체 제품 도달.

### 비즈니스 로직
- 라우팅 = {정보진술 / 추천-구체 / 추천-모호}. 판정은 LLM + 규칙(제형·보습·성분 등 구체 슬롯 존재 여부).
- 근거 생성은 **retrieve가 준 graph_paths·memory_facts만 인용**. 없는 사실 생성 금지(AC-R3).
- 제형은 **soft-ranking**(에멀전>오일 등), 하드 보장 아님. 성분 회피만 하드.
- 퍼널 상태는 서버에 저장하지 않음(**stateless**) — 클라이언트가 턴 히스토리를 전달하고, 퍼널 진행도는 히스토리+기억으로 매 턴 재계산. (durable 사실은 memories에 있음)

### 예외 처리
- 하드필터 후 후보 0 → 빈 추천 대신 대체 방향 제안(F2 예외와 연동).
- 근거로 쓸 경로·기억이 전무 → "정보가 부족하니 몇 가지 여쭐게요"로 퍼널 전환(억지 추천 금지).
- LLM 생성이 회피 성분 포함 제품을 언급하려 함 → 하드필터 단계에서 이미 제거되므로 후보에 없음(생성이 임의로 추가하면 후처리 검증에서 컷).

## 8. 대표 종단 시나리오 (AC-R4, 통합 검증용)

입력: **"가을이 되니 건조해요. 끈적한 제형은 싫어서 오일 말고, 에멀전인데 보습 확실한 거요."**
(전제: 사용자는 과거 "오일 안 맞음"을 기억에 가지고 있음.)

기대 동작(4개 단언 전부 충족해야 완료):
- **(a)** 근거에 **고민→성분 경로 노드**(건조 ←TREATS← 보습 성분)가 포함되고, 사용자 맥락의 **계절(가을)**이 반영된다.
- **(b)** 회피 성분 포함 제품이 결과에 **0건**.
- **(c)** 에멀전 서술 제품이 동일 고민 매칭의 오일 서술 제품보다 **상위**.
- **(d)** **회상된 기억 사실 ≥1개**("오일 회피")가 근거에 등장.

## 9. 비기능 요구 (전역)

- **격리(하드)**: 크로스유저 기억·그래프 조회 0건(AC-M5/G3). 보안 핵심.
- **지연 예산**: 검색(비-LLM) p95 ≤ 400ms, 종단(LLM 포함) p95 ≤ 2.5s(시드 데이터 기준).
- **데이터 출처**: 실공개 데이터/크롤(합성 금지), 출처·수집일시 메타 보존, ToS·rate-limit 준수.
- **다국어**: KO+EN 혼재 입력·문서 처리(bge-m3).
