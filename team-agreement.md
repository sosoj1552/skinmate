# Team Agreement — skinmate 2인 협업 공동 합의본

> 확정일: 2026-07-07 · 근거: [skinmate-consensus-plan.md](skinmate-consensus-plan.md), [open-questions.md](open-questions.md)
> 상태: **고정(FROZEN).** ⭐ 공유 항목을 바꾸려면 반드시 상대와 합의 + PR 리뷰를 거친다.

이 문서는 두 사람이 동시에 개발·실험하기 위한 **역할 분담**과 **함께 지켜야 할 약속(⭐ 공유 항목)** 을 고정한다.

---

## 0. 용어 쉬운말 대응표

낯선 용어가 많아 먼저 풀이한다. (본문에서는 정확성을 위해 원래 이름도 병기)

| 용어                            | 쉬운 뜻                                                           |     |
| ----------------------------- | -------------------------------------------------------------- | --- |
| **⭐ 공유 항목(seam)**             | 두 사람 코드가 **같이 물고 있는 지점**. 여기를 바꾸면 상대가 깨지므로 반드시 합의              |     |
| **데이터 플레인 / LLM 플레인**         | 일감을 나눈 두 영역. 데이터 플레인 = 자료 수집·검색, LLM 플레인 = 기억·대화·평가            |     |
| **임베딩(embedding)**            | 문장을 의미가 담긴 **숫자 벡터**로 바꾼 것. 비슷한 문장은 벡터도 가까움                    |     |
| **pgvector**                  | Postgres에서 그 숫자 벡터로 **유사한 것 찾기**를 해주는 확장 기능                    |     |
| **AGE (그래프)**                 | Postgres 안에서 **관계망(계절→고민→성분→제품)** 을 저장·탐색하는 확장 기능              |     |
| **2-hop 순회**                  | 관계망에서 **두 단계 이상 건너뛰며** 따라가는 것 (예: 사용자→회피성분→그 성분 든 제품 제외→대체 성분) |     |
| **INCI / canonical key**      | 화장품 성분의 **국제 표준 이름/식별자**. 같은 성분을 하나로 합칠 때 쓰는 열쇠                |     |
| **RLS (행 수준 격리)**             | DB가 **"이 사용자 것만"** 보이게 자동으로 막아주는 안전장치                          |     |
| **choke (단일 관문)**             | 그래프에 드나드는 **유일한 통로 함수**. 여기서 "누구 데이터인지"를 강제로 붙여 남의 것과 안 섞이게 함  |     |
| **projection (투영/재구성)**       | 표(관계형) 데이터를 규칙대로 **그래프 연결선으로 다시 만들어내는 것**                      |     |
| **effective_weight (유효 가중치)** | 기억의 **중요도 점수**. 자주·최근 말할수록 높고 시간이 지나면 낮아짐                      |     |
| **원자적 트랜잭션 / rollback**       | 여러 저장을 **한 묶음으로 처리** — 하나라도 실패하면 **전부 취소**해 반쪽 저장을 막음          |     |
| **fixture**                   | 테스트용 **미리 만들어 둔 샘플 데이터**                                       |     |
| **feature 브랜치 / PR**          | 각자 **작업용 가지**에서 개발하고, **합치기 요청(PR)** 으로 본체에 병합                 |     |

## 1. 역할 분담

| 담당 | 맡은 영역 | 소유 폴더 |
|------|-----------|-----------|
| **작업자 A — 데이터 담당** | DB · 자료 수집(크롤링) · 검색 | `db/`, `ingest/`, `knowledge/`, `documents/`, `graph/`, `retrieval/` |
| **작업자 B — 대화·기억 담당** | 기억(개인 사실 저장) · 대화(추천 생성) · 평가(테스트) | `memory/`, `write/`, `chat/`, `llm/`, `app/`, `eval/` |

**약속:** ⭐ 항목은 두 사람이 같이 의존한다 → **바꾸거나 완성하면 반드시 상대와 합의하고 PR 리뷰를 받는다.** 나머지는 각자 자유롭게.

## 2. ⭐ 함께 지킬 공유 항목 (고정)

| ⭐   | 항목            | 무엇을 정했나 (쉬운 설명)                                                                                                 | 주 담당         |
| --- | ------------- | --------------------------------------------------------------------------------------------------------------- | ------------ |
| ⭐9a | 개발 언어         | **Python 3.12** + DB 접속 라이브러리 `psycopg3`                                                                        | 공동           |
| ⭐9b | DB 구성         | **Postgres 16** 하나 안에 그래프(AGE) + 벡터검색(pgvector) 확장 동거. 버전은 정확히 고정                                               | 공동           |
| ⭐9c | 답변 AI         | **Claude API**. 코드에서 직접 부르지 않고 `llm/` 껍데기를 거쳐 부름(나중에 교체 쉽게)                                                     | B            |
| ⭐9d | 임베딩 실행 위치     | 문장→벡터 변환 모델은 **bge-m3**. **처음엔 각자 PC에서 직접 실행, 나중에 컨테이너로.** `embed.py`로 감싸 위치를 바꿔도 코드가 안 바뀌게                     | A            |
| ⭐9e | 협업 방식         | **한 저장소에서 각자 feature 브랜치 → PR로 합치기.** 폴더 소유로 충돌 최소화, ⭐ 파일 변경은 상대 리뷰 필수                                          | 공동           |
| ⭐1a | 테이블 구조        | **[docs/DATA-MODEL.md](docs/DATA-MODEL.md)** 확정본(관계형 6 + 그래프 + 벡터)을 **고정**. `db/migrations/`가 구현 | A            |
| ⭐1b | 구조 변경 방법      | DB 구조 변경은 `db/migrations/` 폴더에 **번호순 SQL 파일**(001_, 002_…)로만                                                    | A            |
| ⭐1c | 그래프 이름        | 관계망 그래프는 하나, 이름은 **`skinmate`**                                                                                 | A            |
| ⭐2  | `embed.py`    | 텍스트를 **1024칸 숫자벡터**로 바꿔주는 변환기. 저장할 때 **어떤 모델로 만들었는지**(`embedding_model_id`)도 같이 기록. 기억 중복 판단도 같은 모델 사용          | A            |
| ⭐3  | `choke.py`    | 그래프에 드나드는 **유일한 통로 함수** `age_exec(...)`. 개인 데이터엔 **"누구 것인지"를 강제로 붙여** 남의 것과 안 섞이게. 다른 곳에서 그래프 직접 호출 **금지**      | A            |
| ⭐4  | `writer.py`   | 기억 저장을 **한 묶음(트랜잭션)** 으로: 표+벡터+그래프를 한 번에 저장, 하나라도 실패하면 **전부 취소**. **답변을 먼저 내보낸 뒤** 저장. 같은 사용자 요청은 줄 세워 처리       | B            |
| ⭐5  | 기억→그래프 다리     | "레티놀 안 맞음" 같은 기억을 그래프의 **회피 연결선**으로 바꿔 넣는 다리. 성분/브랜드/고민 → 연결선, 피부타입 등 → 표에만. 성분 이름을 ID로 못 바꾸면 그래프엔 넣지 않고 로그만 남김 | B(생성)·A(그래프) |
| ⭐6  | `retrieve.py` | 검색 3종(**벡터 유사도 + 그래프 순회 + 기억 순위**)을 하나로 합치는 약속. A의 검색이 B가 만든 **기억 순위 함수** `rank_memory(사용자)` 를 불러 씀             | A(합치기)·B(순위) |
| ⭐7  | `contracts/`  | 두 사람이 주고받는 **데이터 형식** 정의: 성분 참조(`IngredientRef`), 그래프 경로(`GraphPath`), 검색 결과 묶음(`RetrievalContext`)             | 공동           |
| ⭐8  | 테스트 기준·샘플     | 합격 기준 수치는 계획서 §6 그대로. **공용 테스트 샘플**은 `eval/fixtures/`(사용자 2명·제형 라벨 제품·가중치 샘플) — B가 관리, A가 자료 제공                 | B(관리)·A(자료)  |

## 3. 협업 규칙 (feature 브랜치)

1. **폴더 소유 = 1차 충돌 방지선.** 자기 폴더 안에서는 자유롭게 커밋.
2. **⭐ 파일 변경은 PR + 상대 리뷰 필수.** 특히 ⭐1 테이블 구조(`db/migrations/`)와 ⭐7 데이터 형식(`contracts/`).
3. **테이블 구조 고정.** ⭐1 확정 뒤 변경은 새 번호 SQL 파일로만, 상대에게 공지.
4. **먼저 내려줘야 상대가 안 막히는 것(조기 납품):**
   - A는 `choke.py`(⭐3)·`embed.py`(⭐2)의 **껍데기+최소 동작을 초반에 먼저** 내려 B가 기다리지 않게.
   - **소량 공용 샘플 데이터**(⭐8, 성분·제품 조금)를 착수일에 만들어 B의 기억→그래프 다리(⭐5)가 A의 크롤링 완료를 안 기다리게.
5. **커밋·푸시 전 사용자 확인** (프로젝트 규칙).

## 4. 결정에서 따라오는 것 (WBS 반영)

- **⭐9d 임베딩 로컬 우선:** `embed.py`를 처음부터 껍데기로 감싼다 → 나중 컨테이너 전환이 코드 수정 없이. 모델 로딩이 무겁고(Windows CPU) 느리므로 A가 껍데기+최소 동작 먼저 납품, 컨테이너화는 뒤로 분리.
- **⭐9e feature 브랜치:** 두 사람이 만나는 지점(⭐5 다리, ⭐6 검색 합치기, ⭐8 샘플)은 PR에서 부딪히는 곳 → WBS 후반 통합 작업으로 명시.
- **일 순서(임계경로):** `⭐9 개발환경 → ⭐1 테이블 → ⭐7 데이터 형식 → (A: 수집→검색 ‖ B: 기억→저장→대화) → 통합(⭐5·⭐6·⭐8)`.
